# Boxing Memo Product Design v3

## 1. 先に結論

前回までの案より、さらに目的に沿わせるなら、このプロダクトは

"指摘を記録するアプリ"

ではなく

"次の練習で何を直せば最短で伸びるかを毎回決めるアプリ"

として設計した方がよい。

記録、分析、AIは全部そのための手段でしかない。

このプロダクトが毎回ユーザーに返すべき価値は、要約ではなく次の3つ。

1. 今回の練習で意識するのはこれだけ
2. その場で思い出せる短い修正キュー
3. それが改善したかどうかの判定

## 2. 何がまだ弱かったか

前回案にも方向性はあったが、まだ弱い点があった。

### 2.1 記録起点すぎる

構造化やトラッキングはできるが、実際の上達速度を決めるのは

- 練習前に何を持って入るか
- 練習中に何を思い出せるか
- 練習後に何を次回へ持ち越すか

の3点である。

### 2.2 練習中の認知負荷を十分に考慮していない

初心者は練習中に複数のことを同時に修正できない。

だから「未解決の課題一覧」よりも、

- 今回の主課題1つ
- 補助課題1つ

くらいまで削る方が目的に合う。

### 2.3 判定の体験が弱い

"改善しているっぽい" では足りない。

ユーザーが本当に欲しいのは

- 何が改善したのか
- 何が停滞しているのか
- 何を変えれば停滞を破れるのか

である。

### 2.4 AIがまだ裏方すぎる

AIは単に整理役では弱い。

このプロダクトではAIは

- 優先順位を絞る
- 次の一手を出す
- 停滞時に介入を変える

ところまで踏み込むべき。

## 3. 再定義したプロダクトの目的

### 主目的

コーチの指摘を、最短で修正に結びつく実行単位へ変換し、改善の有無を継続判定する。

### 副目的

- 指摘を忘れない
- 同じ指摘の反復を減らす
- 練習ごとのテーマを明確にする
- 自己評価のズレを減らす

### 非目的

以下はMVPでは目的から外す。

- 高精度なフォーム自動判定
- SNS的な共有
- ゲーム化中心の体験
- 一般的な日記機能の充実

## 4. 最適なプロダクト原則

1. 全てを保存するが、画面ではほとんど見せない
2. 毎回の練習で持たせる主課題は1つにする
3. 各課題は必ず「意識」ではなく「行動」に変換する
4. 改善判定には理由を必ず表示する
5. 停滞を検知したら、同じアドバイスを繰り返さない
6. 練習前3分、練習直後30秒で使えるUIにする

## 5. このプロダクトの正しい体験設計

ユーザー体験は「記録」ではなく、4つの瞬間で設計するべき。

### Moment 1: 練習前3分

ユーザーの状態:

- 時間がない
- 集中したい
- 多くの情報はいらない

プロダクトが返すべきもの:

- 今日の主課題1つ
- 補助課題1つ
- それぞれの短いキュー
- 今日コーチに確認すべき一言

ここでは履歴を見せすぎない。

### Moment 2: 練習中の小休止

ユーザーの状態:

- 息が上がっている
- 長文は読めない
- 1秒で思い出したい

プロダクトが返すべきもの:

- 大きい文字のキュー
- 1行の自己チェック
- 今回だけ避けるミス

この瞬間のUIは、実質カンペであるべき。

### Moment 3: 練習直後30秒

ユーザーの状態:

- 疲れている
- ただし記憶は一番新鮮

プロダクトが返すべきもの:

- 音声で一気に入力できる場
- 既存課題の更新候補
- 新課題の候補
- 改善判定の仮説

ここは最速入力が最重要で、フォームの美しさは不要。

### Moment 4: 週次レビュー

ユーザーの状態:

- 少し落ち着いて振り返れる

プロダクトが返すべきもの:

- 今週いちばん改善した課題
- 何度も再発した課題
- 長く止まっている課題
- 来週のテーマ

ここで初めて分析をしっかり見せる。

## 6. 体験の主軸は「Issue」ではなく「Focus Cycle」

前回は Correction や Issue 単位で整理していたが、さらに目的に寄せるなら、中心概念は `Focus Cycle` にした方がよい。

1つの Focus Cycle は以下で構成する。

- 課題
- 原因仮説
- 今回試す修正方法
- 練習中のキュー
- 判定ルール
- 次回更新条件

つまり、単なる問題カードではなく

"今回この直し方で試して、次回この条件で評価する"

という1サイクルを持つ。

これにより、同じ課題でも

- 今の打ち手
- 前回までの打ち手
- なぜ打ち手を変えたか

を管理できる。

これは上達速度に直結する。

## 7. 最も目的に合う情報設計

### 一番上に置くべき情報

1. 今日の主課題
2. 今日の修正キュー
3. 今日の確認項目
4. 前回から何が変わったか

### 二階層目に置くべき情報

1. 補助課題
2. 停滞理由
3. ドリル履歴
4. コーチ発言の原文

### 深い階層に置くべき情報

1. 全ログ
2. 全トレンド
3. 証拠詳細
4. 類似課題の統合履歴

この順番を崩すと、アプリはすぐに「見るだけの記録庫」になる。

## 8. 最適な画面定義

画面は多くなくてよい。むしろ少ない方がよい。

## 8.1 Command Home

この画面はホームではなく、実質「今日の作戦盤」。

表示要素:

- 次回練習までの時間
- 今日の主課題
- 今日の補助課題
- 主課題のキュー
- 主課題の自己チェック
- コーチに聞く一言
- 前回との変化

主CTA:

- `今日の作戦を見る`
- `練習後に記録する`

UI要件:

- 起動して3秒以内に主課題が読める
- 長文禁止
- 数字と色で重要度を示す

## 8.2 Training Card

これは練習中専用の超軽量画面。

表示要素:

- 主課題名
- 超短いキュー
- NG例1つ
- 自己チェック1つ

例:

- 課題: 前のめりにならない
- キュー: 頭ではなく足で入る
- NG: 打った瞬間に顔が前へ出る
- Check: 打った後に重心は真下に戻ったか

UI要件:

- ロック画面感覚で見られる
- スクロール不要
- 片手操作

## 8.3 Quick Debrief

練習直後の入力画面。

入力方法:

- 音声メモ
- 1行テキスト
- 良化 / 横ばい / 悪化 の3択
- 任意で動画添付

AIが返すもの:

- 既存課題への紐付け
- 新課題候補
- 課題の優先度変化
- 次回の主課題案

重要なのは、入力後すぐに「整理された感」が返ること。

## 8.4 Focus Cycle Detail

この画面はその課題を深掘る場。

表示要素:

- 課題名
- いつから続いているか
- 今の原因仮説
- 今回試している修正法
- 過去に試した修正法
- 改善判定の根拠
- 停滞している場合の理由
- 次に変えるべき介入

ここで重要なのは、「履歴」よりも「学習の仮説更新」が見えること。

## 8.5 Weekly Coach Report

これはレビュー画面であり、ダッシュボードではない。

表示要素:

- 今週の主テーマ
- 改善した課題
- 停滞した課題
- 再発した課題
- 来週の集中テーマ
- 無視していい課題

最後の「無視していい課題」はかなり重要。

ユーザーは何を頑張るか以上に、何を今は追わないかを知ると楽になる。

## 9. UIのあるべきトーン

このプロダクトのUIは

- ノートアプリっぽくない
- AIチャットっぽくない
- ガジェットっぽすぎない

方がよい。

狙うべき印象は、

- コーチングボード
- 作戦メモ
- 練習指示カード

である。

### ビジュアル方針

- 背景はフラット白より少し空気感を持たせる
- 主課題だけ強いアクセント色
- 文字サイズの強弱を大きくつける
- チャートは最小限
- カードは「読む」ではなく「指示を受ける」見た目

### 避けるべきUI

- タブが多すぎる
- KPIカードが並びすぎる
- AIの文章が長い
- 毎回チャットさせる

この用途ではチャットUIは主役に向いていない。

## 10. AIの最適な役割

AIの役割は6つに分けた方がよい。

### 10.1 指摘の正規化

表現揺れをまとめる。

例:

- 前のめり
- 突っ込む
- 頭から入る

### 10.2 優先順位の圧縮

全課題の中から

- 今日の主課題
- 今週のテーマ

を決める。

### 10.3 行動への変換

各課題を必ず

- キュー
- ドリル
- 自己チェック
- コーチへの確認文

へ変換する。

### 10.4 停滞検知

同じ課題が続いているのに変化が出ない場合、

- キューを変える
- ドリルを変える
- 原因仮説を見直す

を提案する。

### 10.5 改善判定

改善しているかを、

- コーチ発言
- 自己評価
- 再発頻度
- セッション文脈

から推定する。

### 10.6 ユーザー教育

AIは説明しすぎない範囲で、

- なぜこれが主課題なのか
- なぜ今は別課題を後回しにするのか

を短く説明する。

これにより納得感が生まれる。

## 11. 一番重要なロジックは「停滞時の介入変更」

多くのプロダクトは、課題を記録して同じ助言を繰り返すだけで終わる。

でも、本当に目的に沿うなら、最重要ロジックはこれ。

- 何セッション続いているか
- 同じキューを何回使ったか
- 変化が出ているか
- 別原因の可能性はないか

を見て、停滞したら介入を変える。

例:

課題:

- 前のめりになる

これまでの介入:

- キュー: 頭を突っ込まない
- ドリル: ジャブ後に止まる

4セッション変化なしなら、次は

- 原因仮説変更: 歩幅が大きすぎる可能性
- 新ドリル: 小さいステップ幅での出入り
- 新チェック: 打つ前に前足が流れていないか

と変えるべき。

これが「最短で修正する」に最も効く。

## 12. ステータス設計も少し変えるべき

前回の `new / active / watch / improving / stable / relapsed` は悪くないが、
ユーザーにとってさらに分かりやすくするなら、画面上は以下でよい。

- `今日やる`
- `様子を見る`
- `良くなっている`
- `ほぼ定着`
- `再発`
- `止まっている`

内部的には別の状態管理をしてよいが、表面の言葉は行動を連想させるべき。

特に `止まっている` は重要で、単なる未解決より意味が強い。

## 13. このプロダクトの本当の差別化

差別化は「AI搭載」ではない。

差別化は、

- コーチの言葉を忘れない

ではなく

- コーチの言葉を次回の行動に変え、改善判定まで閉じる

ことにある。

つまり価値は保存ではなく運用にある。

## 14. MVPもさらに絞るべき

本当に強いMVPは次で十分。

1. 練習直後の音声入力
2. AIによる課題抽出と統合
3. 次回の主課題1つと補助課題1つの提示
4. 各課題のキューと自己チェック表示
5. 週次での改善 / 停滞 / 再発レポート
6. 停滞時の介入変更提案

逆に、MVPで不要なもの:

- 動画の自動解析
- 細かいタグ編集
- SNS
- 複雑な目標管理
- 長いAI会話

## 16. スコアリング設計

`main_focus_score` は以下の合成でよい。

- 重要度
- 再発頻度
- 他課題への波及度
- 直近の未改善度
- 今の練習文脈との相性

`resolution_confidence` は以下を使う。

- 指摘頻度の低下
- コーチの言い方の変化
- 自己評価の改善
- 再発有無

`stalled_risk` は以下を使う。

- 同一課題が長い
- 同じ介入を繰り返している
- 改善証拠が増えない
- 同じトリガーで再発している

重要なのは、スコアを出すことより

"なぜその判定なのかを一文で返すこと"

である。

## 17. 理想の1セッション体験

### 練習前

アプリを開くと、

- 主課題: 前のめりにならない
- キュー: 頭ではなく足で入る
- チェック: 打った後に重心は真下か
- コーチに聞く: 今日は前に流れていたらすぐ教えてください

だけが見える。

### 練習後

ユーザーが音声で

"今日は前のめりは前よりまし。代わりに右で肩が開くと言われた"

と残す。

AIが返す:

- 主課題は `良くなっている`
- 理由: 直近2回で指摘が減少
- 新候補: `右で肩が開く`
- 次回方針: 主課題は継続、補助課題に右肩を追加

### 週末

アプリが返す:

- 今週もっとも改善: 前のめり
- 今週もっとも停滞: ガード戻り
- 来週のテーマ: 打った後の姿勢維持

この流れなら、目的にかなり近い。

## 18. 最終的なプロダクト定義

最も目的に沿った定義はこれ。

"A boxing coaching OS that compresses feedback into the next best correction cycle."

日本語なら、

"コーチの指摘を、次回の最適な修正サイクルに変換するボクシング上達OS"

である。
